public class Goods {
    private String name;
    private float price;
    private int quantity;

    static {
        // 静态代码块
        /*
        属于类的一个成员, 会随着类的加载而执行.
        只执行一次.
        会做一些类的初始化工作.
        多个静态代码块之间一般情况下会根据书写顺序, 顺序执行
         */
        System.out.println("这是一个静态代码块");
    }

    {
        // 代码块
        // 属于对象的, 会随着对象的加载而执行
        // 先于构造方法执行
        System.out.println("这是一个代码块");
    }

    public Goods(String name, float price) {
//        this.name = name;
//        this.price = price;
//        this.quantity = 1;

        //super();  这里不能写, 写的话相当于调用了这个super(), 然后再调用3个参数构造方法里的super(), 这样就重复了
        this(name, price, 1);   // 调用3个参数的构造方法
    }

    public Goods(String name, float price, int quantity) {
        super();
        this.name = name;
        this.price = price;
        this.quantity = quantity;
    }
}

内部类

public class Person {
    private String name = "1";
    private  Legs legs;

    public Person() {
        // 在外部类中创建内部类对象
        legs = new Legs();
    }

    public void walk() {
        legs.switchLeg();
    }
    /*
        内部类的使用场景
        当某个类的功能比较复杂, 需要将其拆分为几部分,
        但是在类的外部不需要知道几部分的细节, 那么这个几部分
        就可以声明为内部类.
        内部类的特点, 可以直接引用外部类的所有属性.
        **内部类持有外部类的引用**
         */
    public class Legs {
        private String name = "2";
        public void switchLeg() {
            System.out.println(name + "腿在交换");  // 1腿在交换
            System.out.println(this.name + "腿在交换"); // 1腿在交换
            System.out.println(Person.this.name + "腿在交换");  //2腿在交换
        }
    }

    public static class Inner {
        // 静态内部类
        // 当做是普通类, 外部类就相当于包
    }
}

接口 interface

  1. 接口中的方法都是public的抽象方法, 所以public abstract都可以省略
  2. 接口中的属性都是全局常量, public static final 都可以省略

接口标识的是一种能力, 可以把接口看做是一个特殊的类

抽象类是为了被继承的, 接口是为了被实现的.

继承: extends , 实现: implements

一个类只能直接继承一个类, 但是可以实现多个接口, 多个接口之间使用 逗号 隔开

java中类是单继承的, 但是接口是多继承的, 多态适用于接口

简单计算器
public interface Computable {
    int TYPE_0 = 0;
    int compute(int x, int y);
}

public class AddComputer implements Computable {
    @Override
    public int compute(int x, int y) {
        return x + y;
    }
}

public class SubComputer implements Computable {
    @Override
    public int compute(int x, int y) {
        return x - y;
    }
}

public class MulComputer implements Computable {
    @Override
    public int compute(int x, int y) {
        return x * y;
    }
}

public class DivComputer implements Computable {
    @Override
    public int compute(int x, int y) {
        return x / y;
    }
}

    public static void main(String[] args) {
//        Computable a = new AddComputer();     // 多态适用于接口

//        Computable m = new MulComputer();
//        System.out.println(m.compute(3, 5));
//        Computable d = new DivComputer();

        Scanner sc = new Scanner(System.in);
        System.out.println("请输入第一个整数");
        int x = sc.nextInt();
        System.out.println("请输入第二个整数");
        int y = sc.nextInt();
        System.out.println("请输入运算符");

策略模式

把一个类(A)中经常改变或者将来可能改变的部分提取出来，作为一个接口(B)，然后在类(A)中包含这个接口(B)，这样类(A)的实例在运行时就可以随意调用实现了这个接口的类(C)的行为。比如定义一系列的算法,把每一个算法封装起来, 并且使它们可相互替换，使得算法可独立于使用它的客户而变化。这就是策略模式。

攻击接口
public interface AttackAble {
    void attack();
}
攻击接口的具体实现类,用刀攻击
public class KnifeAttack implements AttackAble {
    @Override
    public void attack() {
        System.out.println("用刀攻击");
    }
}
英雄类
public abstract class Hero {
    protected AttackAble attackAble;    // 注意不能是private
    protected MoveAble moveAble;        // 接口属性, 将类和接口连接起来, 用于在子类的构造方法中实例化接口实现类的对象

    public abstract void attack();

    public void setAttackAble(AttackAble attackAble) {
        this.attackAble = attackAble;
    }
}
英雄类的子类战士类
public class Warrior extends Hero {
    public Warrior() {  // 构造方法, 用于实例化接口的具体实现类的对象
        AttackAble attackAble = new KnifeAttack();
        this.attackAble = attackAble;
    }
    @Override
    public void attack() {
        attackAble.attack();
    }
}

最少知道原则(迪米特原则)

一个实体应当尽量少地与其他实体之间发生相互作用，使得系统功能模块相对独立。通俗地说就是不要和陌生人说话，即一个对象应对其他对象有尽可能少的了解。迪米特法则的初衷在于降低类之间的耦合。由于每个类尽量减少对其他类的依赖，因此，很容易使得系统的功能模块功能独立，相互之间不存在（或很少有）依赖关系。

类与类之间尽量少建立关系
